## 1. 产品定义

Sift 是一款桌面端通用数据采集工具,采用「双入口、单引擎、多输出」架构。

**双入口** —— 提供两种定义采集目标的方式:

- 入口 A — 可视化点选:加载真实网页,通过鼠标悬停高亮、点击选中元素,自动生成选择器。面向不具备编码能力的用户。
- 入口 B — 规则配置 / 导入:粘贴 JSON 采集规则(兼容小说"书源"DSL 格式),系统解析、识别并可视化展示。面向高级用户与复杂多步抓取场景。

**单引擎** —— 两个入口最终均编译为统一的内部规则中间表示(Rule IR),由同一采集引擎执行。

**多输出** —— 表格预览 + 批量下载(文本 / 图片 / 视频),并支持自动化(定时调度 / 数据变化通知)。

**技术栈**:Vite + Tauri 2.x + Vue 3 + TypeScript;pnpm monorepo;采集核心由 Rust 实现。

**产品目标**:不具备编码能力的用户可通过可视化点选采集数据,高级用户可通过粘贴 JSON 规则快速理解结构。整体定位为本地优先、中立、专业级的采集工具。

## 2. 产品定位与合规边界

本节为产品底线,所有功能设计须遵守。

**核心原则:工具提供「能力」,不预置「目标」。**

工具保持中立:用户使用自身身份与凭据,采集自身有权访问的数据,内容责任归于用户(类比浏览器、curl、Postman)。预置特定站点的破解规则会使责任转移至产品方,因此禁止。

### 合规分级

| 级别 | 范围 | 处置 |
|---|---|---|
| 安全标配 | 自定义选择器、自定义请求头(含 Cookie)、自定义 UA、限速、代理、批量下载 | 实现,采用中立工具方式 |
| 谨慎设计 | 浏览器导入 Cookie、接口嗅探、用户 JS 脚本、规则分享 | 实现,附加安全 / 审核设计 |
| 禁止 | 模拟登录(填账密自动登录)、自动绕过验证码、DRM 解密、内置盗版源、撞库 | 不实现 |

### 凭据处理原则(Cookie / Header)

1. 界面明确提示:仅填入用户有权访问的账号凭据
2. 凭据本地加密存储,不明文落盘,不离开本机
3. 不预置任何账号,不在用户间共享凭据

## 3. 用户画像

| 维度 | 核心用户 A:技术运营 / 数据分析师 | 核心用户 B:中小企业主 / 电商卖家 | 边缘用户 C:开发者 |
|---|---|---|---|
| 技术水平 | 掌握简单 XPath/CSS,不一定具备编码能力 | 几乎不具备技术背景 | 熟悉 HTTP / 正则 / 脚本 |
| 核心诉求 | 定期抓取竞品价格、舆情、表格数据 | 监控店铺评论、价格、库存 | 快速验证抓取逻辑、导出脚本 |
| 主要痛点 | 运行环境配置复杂、反爬处理困难、调度繁琐 | 不具备编码能力、担心账号风险 | 重复开发、缺乏可视化调试 |
| 主要入口 | 规则配置 + 调试 | 可视化点选 | 规则 + 代码导出 |
| 付费意愿 | 中 | 高 | 低 |

设计取向:渐进式暴露复杂度。基础用户界面保持简洁(点选 + 字段),高级能力(请求头 / 正则 / 脚本)以折叠形式提供。

## 4. 功能范围与分期

### 第一阶段(MVP)

| 模块 | 功能 |
|---|---|
| 规则定义 | 自定义选择器(CSS / XPath / JSONPath 三模式)、点选与手写双向同步、正则提取 |
| 可视化点选 | 网页预览、悬停高亮、点击选中、重复结构识别(点选一个匹配整列)、选择器稳健性评分、属性提取(懒加载 data-src) |
| 规则导入 | 粘贴 JSON、解析识别(API / 网页源)、结构化卡片展示、Base64 解码、指令剥离、多步骤链路可视化、未知字段容错 |
| 请求控制 | 自定义请求头 / Cookie(加密存储)、自定义 UA、限速 / 并发(默认开启)、超时 / 重试 |
| 抓取模式 | 静态 HTTP 抓取、接口直连(JSON) |
| 调试 | 调试控制台:逐步执行采集链路,显示每步请求 / 响应 / 解析结果 |
| 输出 | 表格预览、导出 CSV / JSON / Excel、批量下载文本(TXT) |

### 第二阶段

代理设置(HTTP / SOCKS5)、动态渲染抓取、接口嗅探(监听 XHR/Fetch 定位数据来源)、定时调度(cron)、增量抓取(指纹去重)、数据变化通知、代码导出(转 Python / JS)、图片批量下载(ZIP / PDF / CBZ,含防盗链与懒加载处理)、导出 EPUB。

### 第三阶段

浏览器导入 Cookie、内嵌 JS 处理脚本(沙箱执行)、视频下载(m3u8 / HLS 解析 + ts 合并 + 可选 FFmpeg)、Webhook / 导出至数据库、规则市场(含审核机制)。

### 不实现

模拟登录、自动绕过验证码、DRM 解密视频、内置盗版源。

│  │  ·静态 HTTP        ·CSS+伪类       ·正则清洗       │   │

└────────────────────────────────────────────────────────────┘

### 5.2 关键架构决策

**① 统一规则 IR 为架构核心。** 点选、书源、手写规则三种输入编译为同一 `Rule` 数据结构,引擎仅识别 IR,与输入来源解耦。新增输入方式只需新增转译器,引擎不变。此为"双入口单引擎"成立的基础。

**② 能力插件化。** 解析层(CSS / XPath / JSONPath)为可替换 parser,按规则的 `engine` 字段分发;下载器(文本 / 图片 / 视频)为实现同一 `Downloader` trait 的模块,共用调度层。视频下载于第二、三阶段插入,不影响主体。

**③ 采集核心置于 Rust。** 理由:并发性能、绕过浏览器同源 / CORS 限制、可控 TLS 指纹、加密凭据存储。前端仅负责交互与展示。

**④ 安全隔离。** 因引入 Cookie 与用户 JS:凭据使用 Tauri `stronghold` 或系统 keychain 加密,不明文存储;用户 JS 在独立沙箱执行,剥夺文件 / 网络权限,仅处理传入字符串。

### 5.3 Monorepo 包划分

```

```

| 用途 | 选型 | 备注 |

### 6.3 线 B 验证项(书源解析)

逐项编写测试断言:

1. **源类型识别**:`source_type:"2"` 判定为 API 源(JSON);无 type 但含 CSS 选择器判定为网页源;含 `key$$` 倾向 API
2. **JSON 路径归一**:`key$$data.key$$search_list` 解析为 `data.search_list`
3. **前缀指令剥离**:从 `{{302}}{{gb2312}}http://...` 分离指令数组与纯 URL;`302` 标记跟随重定向,`gb2312` 标记编码
4. **占位符提取**:`###keyword###`、`###book_id###`、`###item_id###`
5. **多步骤链路构建**:将 search_url → search_result.url → book_menu → item_url → chapter_content 组织为有序步骤链,每步记录 URL、占位符依赖、规则、解析模式
6. **content_filter Base64 解码**:检测并解码为可读文本,失败则原样保留并标记
7. **jQuery 伪类转译**(标准 CSS 不支持):`:gt(N)` 转 slice(N+1)、`:lt(N)` 转 slice(0,N)、`:eq(N)` 转 nth(N)。测试:15 元素列表 `:gt(8)` 返回后 6 个
8. **多选择器 fallback**:逗号分隔解析为备选数组,依次尝试命中即止
9. **url_replace_rules**:`http://m%%http://www` 以 `%%` 分隔解析为替换规则
10. **未知字段容错**:不在已知字段表的 key 收入容错集合,不报错
11. **完整性校验**:缺关键字段产出 warnings 而非崩溃,输出状态 ok / warning / error

**HTML 解析库选型**:网页源需 CSS 选择器解析,标准 querySelector 不支持伪类。评估 `node-html-parser` / `cheerio` + 自实现伪类层,在 REPORT 说明选型理由。伪类转译逻辑须独立可测,不依赖具体 DOM 库即可验证核心算法。

### 6.4 线 A 验证项(可视化点选)

注入交互层不在验证范围,仅验证算法。逐项编写测试:

1. **稳健选择器生成**:给定 DOM 节点生成唯一 CSS 选择器。优先级:有意义的 `id` > 语义 `class` > `nth-of-type`。须跳过随机值(如 `id="item-a3f9b2c1"`、纯数字 id)
2. **重复结构识别(核心)**:给定选中的一个元素,向上回溯找到含多个结构相似兄弟的祖先层(列表容器),推断列表项。返回容器、列表项标签 / class、项数、项内到目标的路径
3. **从一个到整列**:基于重复结构生成匹配所有同类元素的列表选择器。测试:5 个结构相同的商品卡,点选第 3 个的标题,生成的选择器匹配全部 5 个标题
4. **多字段联动**:同一列表下点选标题 / 价格 / 图片分别生成各自列选择器,均基于同一列表容器
5. **属性提取识别**:选中 `<a>` 提示可取 `@href`;选中 `<img>` 且真实地址在 `data-src`(懒加载)提示取 `@data-src`
6. **稳健性评分(可选)**:为生成的选择器评分(用 id / 语义 class 得高分,用 nth-of-type 得低分),用于界面提示

**测试数据(放入 fixtures.ts)**:构造电商列表页 DOM 树 —— `body > header + div.product-list#list(内含 5 个 div.product-card,每张含 a.title[href] + span.price + img.cover[data-src]) + footer`。以纯对象模拟 DOM(含 parent 引用),不依赖真实浏览器。

### 6.5 验证报告要求

每个验证产物填写 REPORT.md,包含:

1. 每个验证项的结论(通过 / 警告 / 失败)
2. 关键输出示例(结构化展示)
3. HTML 解析库选型结论与理由
4. 遗留风险(下方第 6.6 节列出的真实环境项)

### 6.6 待真实环境验证项

以下项无法在本阶段覆盖,须在对应阶段验证,在 visual-picker/REPORT.md 中列为遗留风险:

| 风险 | 说明 | 验证阶段 |
|---|---|---|
| WebView 脚本注入 | 点选需向目标页注入 JS 监听悬停 / 点击,部分站点 CSP 会拦截 | Tauri 集成阶段 |
| 真实编码 / 反爬 | GB2312 解码、302 跟随在真实请求中的表现 | 引擎集成阶段 |
| 动态渲染体积 | 复用 WebView 或内置 Chromium | 第二阶段预研 |
| 伪类执行库 | 确认 `node-html-parser` + 转译层或替换库 | 引擎实现阶段 |

**降级方案**:若 WebView 注入被 CSP 拦截,采用"抓取 HTML 快照点选"模式——后端抓取静态 HTML,前端在快照上点选,不依赖注入。

### 6.7 验证通过判据

- 两条线全部验证项测试通过
- 两份 REPORT 完成,选型结论明确,遗留风险已登记
- types.ts 完整,供后续 UI 复用

### 8.1 总体阶段

                 (MVP)         (MVP)          (MVP)           (二期)        (三期)

### 8.2 里程碑明细

**M0 — 技术预研与工程地基**

- [ ] 完成第 6 节技术验证,产出两份 REPORT
- [ ] WebView 注入真实环境验证(Tauri 起壳后)
- [ ] 工程脚手架:Vite + Tauri + Vue + TS,pnpm monorepo
- [ ] 工程化配置:prettier / eslint / stylelint / vitest / CI
- [ ] 定义 core-ir:Rule IR 类型(前后端共享)

出口标准:脚手架可运行,Rule IR 类型定稿,两条验证代码收编为正式包。

**M1 — 采集引擎核心(MVP)**

- [ ] 请求层:静态 HTTP + 自定义 header / Cookie + 限速 / 重试 / 超时 + 编码处理(GB2312)+ 302
- [ ] 解析层:CSS(含伪类转译)+ JSONPath + XPath
- [ ] 多步骤调度器:变量传递 + 翻页
- [ ] 管线层:正则清洗 + Base64
- [ ] 凭据加密存储
- [ ] 用两个真实书源端到端跑通(本阶段允许真实请求,小范围、限速、自有可访问数据)

出口标准:给定 Rule IR,引擎完整执行并返回结构化结果。

**M2 — 双入口 UI(MVP)**

- [ ] 应用外壳:左导航(分组式)+ 顶栏 + 底部状态栏
- [ ] 入口 B 规则导入工作台:粘贴 → 解析卡片(源类型徽章、三步链路图、Base64 解码展示、伪类 tooltip、容错区)
- [ ] 入口 A 点选工作台:网页预览 + 悬停高亮 + 点击选中 + 字段卡(选择器、稳健性评分、属性提取)
- [ ] 自定义选择器输入 + 点选与手写双向同步
- [ ] 调试控制台:逐步执行,显示每步请求 / 响应 / 解析

出口标准:两个入口均产出 Rule IR 并交引擎执行,调试可视。

**M3 — 输出与下载(MVP)**

- [ ] 数据表格预览(虚拟滚动,缩略图列)
- [ ] 导出 CSV / JSON / Excel
- [ ] 下载调度层:队列 / 并发 / 限速 / 重试 / 断点续传 / 进度回传
- [ ] 文本下载器(TXT,章节合并)

出口标准:MVP 主链路闭环——定义规则 → 运行(实时进度)→ 预览 → 导出 / 下载。

**M4 — 自动化与扩展(第二阶段)**

代理设置、动态渲染、接口嗅探、定时调度、增量抓取、数据变化通知、代码导出、图片下载器、EPUB 导出。

**M5 — 进阶能力(第三阶段)**

浏览器导入 Cookie、用户 JS 脚本(沙箱)、视频下载器、Webhook / 数据库导出、规则市场(审核机制)。

### 8.3 关键路径与依赖

```
core-ir (M0) ──┬──> 引擎 (M1) ──> 下载器 (M3)
               ├──> 点选工作台 (M2)
               └──> 规则工作台 (M2)
```

- core-ir 为所有后续工作的前置依赖,M0 须先定稿,否则前后端无法并行
- 引擎(M1)与 UI 外壳(M2 部分)可在 IR 定稿后并行
- 下载器(M3)依赖引擎产出结构化数据

### 8.4 主链路优先级

主链路「定义规则 → 运行(实时进度)→ 表格预览 → 导出 / 下载」须保持流畅,为产品核心价值证明点,任何阶段不得牺牲其流畅度。

## 12. 设计稿覆盖说明与后续功能界面规划

### 12.1 设计稿覆盖策略

界面设计采用分级策略,避免过早设计导致返工:

- **MVP 核心界面**:点选采集、规则导入、请求配置、调试控制台。高频使用、信息架构复杂,产出高保真设计稿作为开发定稿。
- **MVP 列表 / 结果界面**:任务列表、下载队列、已完成、数据预览 / 下载。技术模式常规、需求稳定,产出设计稿。以列表 / 队列 / 表格为主,彼此同构,统一这一类页面的设计语言。
- **MVP 系统界面与引导**:设置、凭据管理、日志、帮助、首次启动引导。技术确定、需求稳定,产出设计稿。
- **第二、三阶段界面(文字规划 + 待验证后再设计)**:定时任务、通知规则,及其余二三期功能。这些界面依赖技术验证结果与实际需求,在进入对应里程碑前,先由技术验证确认可行性与能力边界,再依据可实现范围产出设计稿。**不在 MVP 阶段产出完整设计稿,以避免需求与技术变化导致返工。**

至此 MVP 范围内全部界面已产出设计稿;第二、三阶段界面按下方文字规格规划,进入对应阶段时依第 14 节指南补充设计。

**原则:界面设计须在对应功能的技术可行性确认之后进行,确保所画即所能实现。**

### 12.2 后续界面文字规格(第二、三阶段)

MVP 范围界面均已产出设计稿。以下为第二、三阶段界面规格,统一套用既有框架(左导航分组 + 三区布局 + 深色靛蓝紫视觉),进入对应里程碑时依第 14 节指南设计。

**代理设置(第二阶段)**

- 类型(HTTP / SOCKS5)+ 地址 + 端口 + 可选认证;可保存多组并按任务关联

**动态渲染配置(第二阶段)**

- 切换静态 / 动态抓取模式;动态模式下设置等待条件(元素出现 / 固定延时)、滚动加载

**接口嗅探(第二阶段)**

- 加载页面时列出捕获的 XHR / Fetch 请求(类似网络面板),展示 URL / 方法 / 响应预览
- 用户可从某个接口直接生成采集规则;依赖技术验证确认 CDP 捕获可行性

**定时调度(第二阶段)**

- 可视化频率选择(每日 / 每周 / cron 表达式)+ 任务关联
- 明确告知需保持应用运行 / 开机自启

**增量抓取与数据变化通知(第二阶段)**

- 增量:指纹去重配置;通知:变化触发条件 + 通知方式(系统通知)

**代码导出(第二阶段)**

- 将规则导出为 Python / JS 脚本;预览生成代码

**用户 JS 脚本(第三阶段)**

- 字段后处理脚本编辑器;沙箱执行,剥夺文件 / 网络权限;依赖技术验证确认沙箱方案

**视频下载(第三阶段)**

- 解析 m3u8 / HLS,选择清晰度,展示分片下载进度与合并状态
- 依赖技术验证确认 m3u8 解析、ts 合并、加密流处理与 FFmpeg 依赖的能力边界;DRM 不实现

**规则市场(第三阶段)**

- 规则浏览 / 导入 / 分享;含审核机制防止盗版源分发

## 13. 高风险功能可行性验证(进入对应阶段前执行)

第二、三阶段含若干高不确定性功能,须在投入开发或设计前完成专项可行性验证,确认能力边界。验证方法与第 6 节一致:聚焦核心技术点,不对真实站点发起破坏性请求。

| 功能 | 验证目标 | 验证内容 | 阶段 |
|---|---|---|---|
| 接口嗅探 | CDP 能否稳定捕获网络请求 | 用 `chromiumoxide` 加载页面,捕获 XHR / Fetch 的 URL、方法、响应,确认覆盖率 | 二期前 |
| 动态渲染 | 复用 WebView 或内置 Chromium | 对比 Tauri WebView 与 `chromiumoxide` 的渲染能力、体积、资源占用 | 二期前 |
| 视频下载 | m3u8 链路能否打通 | `m3u8-rs` 解析分片 → 顺序合并 ts → 确认 AES-128 解密可行性、FFmpeg 依赖必要性 | 三期前 |
| 用户 JS 沙箱 | 能否安全执行用户脚本 | 评估 `boa` / `quickjs` / WebView 隔离,验证文件 / 网络权限可被剥夺 | 三期前 |
| 浏览器 Cookie 导入 | 能否读取并解密浏览器 Cookie | 验证读取浏览器 cookie 库 + 平台级解密(keychain / DPAPI)可行性 | 三期前 |

每项验证产出结论:可行 / 受限可行(注明边界)/ 不可行。设计与开发依据验证结论调整范围。

## 14. 后续设计稿 Prompt 编写指南

进入新里程碑、需为某功能产出设计稿时,按本指南编写设计 Prompt,确保新界面与既有设计保持一致,且不偏离已确认的技术边界。

### 14.1 编写前置条件

1. 该功能的技术可行性已确认(MVP 功能已验证;二期三期功能须先完成第 13 节对应验证)
2. 已明确该功能的能力边界(哪些做、哪些受限、哪些不做),设计须在此边界内
3. 已确定该界面在导航结构中的位置(归属哪个分组)

### 14.2 Prompt 必含要素

每份设计 Prompt 须包含以下部分:

1. **风格一致性声明(置于开头,必写)**
   - 明确要求延续既有 Sift 视觉系统,不得另起风格
   - 复述核心视觉规范:深色背景(主背景 `#0E0E14`,侧栏 `#0A0A10`,卡片 `#16161E`)、靛蓝紫强调色 `#7C5CFC`、技术内容用等宽字体、卡片圆角 10–12px、极细边框、几乎无阴影
   - 要求复用既有组件语言:左导航分组式、紫色高亮胶囊、灰底 chip 标签、三级按钮、底部状态栏
2. **界面定位**
   - 该界面属于哪个导航分组、解决什么场景、面向哪类用户
3. **布局结构**
   - 给出分区示意(参照既有屏的三区 / 左右分栏 / 步骤流等模式),优先复用既有布局范式
4. **真实数据要求**
   - 使用真实示例数据填充,不使用占位假数据;若涉及采集规则,沿用既有示例(七猫 API 源 / 旧钢笔网页源)保持连贯
5. **能力边界约束(关键)**
   - 明确写出该功能"做什么、不做什么",防止设计出技术上无法实现或已决定不实现的元素
   - 例:视频下载界面须注明"不含 DRM 内容,加密流处理以验证结论为准"
6. **状态覆盖**
   - 要求覆盖空态 / 正常态 / 错误态 / 加载态(按界面适用情况)
7. **交付画面清单**
   - 按重要性列出需交付的具体画面

### 14.3 风格一致性声明模板
